# Руководство тренера по приложению «Штаб»

«Штаб» — это личная админка тренера для ведения клиентской базы, контроля пропавших клиентов и пятничного ритуала: посмотреть, кому пора написать, собрать строки для Claude, отправить сообщения лично и отметить касания.

Главный принцип MVP: система не отправляет сообщения за тренера и не заменяет личный контакт. Она помогает вовремя заметить клиента и подготовить черновик.

## 1. Быстрый старт

1. Открой приложение.
2. Если база пустая, зарегистрируй первого тренера через `/register`.
3. В дальнейшем заходи через `/login`.
4. После входа основная навигация доступна в боковом меню на компьютере и в нижней панели на телефоне:
   - `Сегодня`
   - `База`
   - `Панель`
   - `Настройки`

Если приложение открывается локально, технический запуск остаётся в `README.md`: сначала Postgres, потом миграции, потом dev-server.

## 2. Основной еженедельный сценарий

Рекомендуемый ритуал раз в неделю, например в пятницу:

1. Открой `База` и обнови клиентов, у которых была тренировка, новый статус или важная заметка.
2. Открой `Сегодня`.
3. Выбери клиентов с триггерами.
4. Нажми `Скопировать для Claude`.
5. Вставь текст во внешний чат Claude.
6. Отправь сообщения клиентам лично через Telegram, SMS или другой канал.
7. После отправки отметь касание в `Сегодня` или в карточке клиента.
8. Открой `Панель` и заполни строку недели.

Цель ритуала: не держать базу в голове и не терять людей молча.

## 3. Экран «Сегодня»

`Сегодня` показывает клиентов, которым пора написать. Триггеры считаются автоматически по статусу клиента, дате последнего касания и настройкам порогов.

Группы триггеров:

| Группа | Что значит | Что делать |
|---|---|---|
| `Тихие` | Клиент давно без касания. Это страховочный триггер от полной потери контакта. | Разобрать первым: написать, позвонить или перевести в `Ушёл`, если клиент действительно закрыт. |
| `Срочно` | Лид или активный клиент уже просрочен по порогу. | Написать в ближайший рабочий слот. |
| `Средне` | Есть повод напомнить, но ситуация не критичная. | Включить в текущий пятничный обход. |
| `Можно подождать` | Остывшие клиенты, которых полезно возвращать мягко. | Писать, если есть время и нормальный повод. |

Действия на экране:

- `Выбрать все` — выделяет все текущие триггеры.
- Чекбокс у клиента — включает его в экспорт для Claude.
- `Скопировать для Claude` — собирает промпт из настроек и выбранные строки без контактов и телефонов.
- `✓ Отметил` — быстро записывает касание типа `Сообщение` и убирает клиента из списка.
- Клик по имени — открывает карточку клиента.

Важно: если у клиента не заполнен `Личный факт`, приложение предупредит об этом. Без личного факта черновик сообщения хуже и может звучать шаблонно.

## 4. База клиентов

`База` — главный справочник клиентов и лидов. Здесь можно искать, фильтровать, добавлять клиентов, добавлять лидов и импортировать CSV.

### Карточка клиента

В карточке хранятся:

| Поле | Для чего нужно |
|---|---|
| `Имя` | Основная идентификация клиента. |
| `Контакт` | TG-ник или телефон. В экспорт для Claude не попадает. |
| `Статус` | Влияет на триггеры. |
| `Профиль` | Помогает понимать мотивацию: здоровье, форма, энергия. |
| `Источник` | Нужен для панели и понимания, откуда приходят клиенты. |
| `Личный факт` | Нужен для персонального сообщения через Claude. |
| `Цель` | Контекст для тренировок и сообщений. |
| `Тренировок/неделю` | Рабочая частота занятий. |
| `Последняя тренировка` | Важна для активных клиентов. |
| `Бронь на сентябрь` | Важна для клиентов в отпуске. |
| `Заметка` | Любой свободный контекст. |

Статусы:

- `Лид` — человек ещё не стал клиентом; если долго нет касания, появится триггер.
- `Предзапись` — есть будущая договорённость.
- `Активный` — текущий клиент.
- `Отпуск` — клиент временно не занимается; важно понимать, есть ли бронь.
- `Остыл` — контакт был, но клиент не активен.
- `Ушёл` — закрытый клиент; триггеры по нему не появляются.

### Касания

Касание — это запись, что контакт с клиентом состоялся.

Типы касаний:

- `Сообщение`
- `Звонок`
- `Тренировка`
- `Другое`

После касания триггер пересчитывается. Если это был единственный повод написать, клиент исчезнет из `Сегодня`.

## 5. Импорт CSV

Импорт нужен для переезда из Excel или другой таблицы.

Как пользоваться:

1. Открой `База`.
2. Перейди в импорт.
3. Перетащи CSV-файл или выбери его через окно файла.
4. Проверь превью и ошибки.
5. Если ошибок нет, подтверди импорт.

Требования:

- формат `.csv`;
- до 5000 строк;
- поддерживаются UTF-8 и Windows-1251;
- ожидается структура из 15 колонок;
- импорт не должен записывать частично: если есть критичные ошибки, сначала исправь файл.

Практика: перед первым большим импортом лучше сделать пробный CSV на 3-5 строках и проверить, что статусы, даты и источники распознаны правильно.

## 6. Панель

`Панель` нужна для еженедельной управленческой строки: откуда пришли заявки, сколько было пробных, сколько стало постоянными и какая загрузка.

Заполняемые поля недели:

| Блок | Поля |
|---|---|
| Лиды по источникам | `Ресепшн`, `Лифты`, `Авито`, `Сарафан`, `База`, `Чат` |
| Тренировки | `Пробные`, `Новые постоянные`, `Загрузка %` |
| Контекст | `Заметка` |

Зачем это делать:

- видеть, какие каналы реально дают людей;
- не путать ощущение с фактами;
- отслеживать, живёт ли пятничный ритуал;
- понимать, когда база начинает проседать.

Если несколько недель подряд строка не заполняется, это сигнал: либо инструмент не встроился в работу, либо сам ритуал нужно упростить.

## 7. Настройки

`Настройки` управляют двумя вещами: промптом для Claude и порогами триггеров.

### Промпт для Claude

Промпт добавляется перед выбранными клиентскими строками из `Сегодня`. Его можно менять под свой голос.

Хороший промпт должен:

- просить короткие личные сообщения;
- сохранять человеческий тон;
- использовать имя и личный факт;
- не обещать лишнего;
- завершать сообщение вопросом или мягким приглашением.

### Пороги триггеров

Настраиваемые пороги:

| Поле | Дефолт | Что означает |
|---|---:|---|
| `Лид без касания` | 3 | Через сколько дней лид без касания становится срочным. |
| `Активный: средний` | 10 | Первый сигнал, что активный клиент давно без тренировки/касания. |
| `Активный: срочный` | 21 | Срочный сигнал по активному клиенту. |
| `Остывший` | 30 | Когда стоит мягко вернуть остывшего клиента. |
| `Тихий` | 45 | Последняя страховка: любой не ушедший клиент давно без касания. |

Не стоит часто менять пороги. Лучше прожить 2-4 недели на дефолтах, посмотреть поведение базы и только потом подкрутить.

### Смена пароля

Для смены пароля введи:

1. текущий пароль;
2. новый пароль;
3. повтор нового пароля.

После успешной смены текущая сессия остаётся активной.

## 8. Рекомендованный рабочий режим

Ежедневно:

- добавлять новых лидов сразу после контакта;
- отмечать реальные касания;
- обновлять дату последней тренировки у активных клиентов.

Раз в неделю:

- открывать `Сегодня`;
- выгружать выбранных клиентов в Claude;
- отправлять сообщения вручную;
- отмечать касания;
- заполнять `Панель`.

Раз в месяц:

- просмотреть `Остыл` и `Ушёл`;
- почистить заметки;
- проверить, какие источники дают постоянных клиентов;
- оценить, не нужно ли поправить пороги.

## 9. Ограничения текущего MVP

Сейчас приложение уже закрывает основной цикл, но важно понимать границы:

- нет автоотправки сообщений;
- нет встроенного LLM-вызова;
- нет календаря тренировок;
- нет платежей и абонементов;
- нет push/email-напоминаний;
- нет многопользовательской модели для команды;
- контакты не уходят в экспорт для Claude, но сама база всё равно требует аккуратного обращения с персональными данными.

Эти ограничения осознанные: MVP должен быть лёгким инструментом для одного тренера, а не большой CRM.

## 10. Перспективы доработки

### Ближайшие полезные фичи

1. **Приём лидов с лендинга**
   - POST `/api/leads` с токеном.
   - Квиз или форма на лендинге сразу создаёт лида в базе.
   - Польза: меньше ручного переноса и меньше потерянных заявок.

2. **Встроенная генерация черновиков**
   - Кнопка не только копирует строки, а сразу вызывает LLM API.
   - Тренер редактирует готовые черновики в интерфейсе.
   - Важно сохранить человека в контуре: отправка всё равно вручную.

3. **Напоминание о пятничном ритуале**
   - Email, web-push или простое уведомление.
   - Срабатывает, если строка недели не заполнена или есть срочные триггеры.

4. **Счётчик экспортов**
   - Записывать факт «экспорт для Claude сделан».
   - Поможет понять, живёт ли ритуал, даже если касания отмечаются не идеально.

5. **Улучшение импорта**
   - Сохранение шаблона CSV.
   - Более подробная подсказка по ошибкам.
   - Предпросмотр того, какие клиенты будут добавлены, а какие обновлены.

### Среднесрочные фичи

1. **Когортная аналитика**
   - Смотреть клиентов по месяцу прихода.
   - Понимать, какие источники дают удержание, а не только заявки.

2. **История изменения статусов**
   - Когда клиент стал активным, остывшим или ушёл.
   - Полезно для анализа, но не обязательно для первого рабочего цикла.

3. **Более умные триггеры**
   - Отдельные пороги по профилям.
   - Отдельные правила для лидов из разных источников.
   - Приоритет по потенциальной ценности клиента.

4. **Шаблоны сообщений**
   - Несколько промптов: лиды, остывшие, активные, отпуск.
   - Быстрое переключение в `Настройках`.

5. **Мобильные улучшения**
   - Ещё более быстрый сценарий «открыл клиента → отметил касание».
   - Улучшенная работа одной рукой.

### Большие фичи, которые стоит делать только после проверки спроса

1. **Мультитенантность**
   - Несколько тренеров, отдельные базы, роли.
   - Это уже не личный инструмент, а продукт.

2. **Биллинг и тарифы**
   - Нужны только если «Штаб» превращается в SaaS.

3. **Интеграции с мессенджерами**
   - Потенциально удобно, но рискованно: можно сломать личный тон и получить проблемы с доставляемостью.

4. **Расписание, абонементы, платежи**
   - Делать только если точно понятно, что это не дублирует ресепшн или другую систему.

## 11. Что не стоит добавлять без отдельного решения

- Авторассылку от имени тренера.
- Большой kanban продаж.
- Сложный скоринг лидов.
- Теги без ограничений.
- Две параллельные системы расписания.
- Финансовый учёт, если платежи уже ведутся в другом месте.

Каждая такая фича увеличивает вес системы. Для текущей задачи важнее, чтобы тренер открывал приложение каждую неделю и реально закрывал хвосты по клиентам.

## 12. Критерий, что MVP работает

MVP можно считать рабочим, если:

1. Клиентская база заведена вручную или импортирована из CSV.
2. На `Сегодня` появляются понятные триггеры.
3. Тренер раз в неделю копирует строки для Claude.
4. Сообщения отправляются лично.
5. Касания отмечаются после отправки.
6. В `Панели` заполняется строка недели.
7. Через 2-4 недели видно, какие клиенты возвращаются и какие источники дают результат.

Если эти пункты выполняются, приложение уже приносит пользу: оно снижает риск тихой потери клиентов и превращает работу с базой в повторяемый ритуал.
