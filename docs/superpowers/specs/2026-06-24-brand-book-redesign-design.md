# Редизайн «Штаба» по брендбуку v2.0

**Дата:** 2026-06-24  
**Статус:** утверждено  
**Источник визуальных правил:** `BRAND-BOOK-updated.html`  
**Исходник знака:** `public/favicon.png`

## 1. Цель

Привести весь интерфейс `athome-admin` к актуальному брендбуку «Тренер у дома» v2.0. Текущая бизнес-логика, маршруты, серверные действия и структура данных сохраняются. Меняются визуальная система, представление семантических состояний и первая сторона контента, где ещё используются emoji.

Выбранное направление — **вариант B, «Неоновая витрина»**:

- крупнее заголовки;
- больше воздуха между смысловыми блоками;
- глубокие стеклянные поверхности;
- заметное, но контролируемое cyan/violet-свечение;
- более выраженная связь с публичным брендом;
- сохранение достаточной плотности для ежедневной работы.

## 2. Утверждённые решения

### 2.1. Логотип

- `public/favicon.png` — эталон формы знака: контур дома и монограмма NK.
- Этот PNG используется как favicon вкладки браузера.
- Для интерфейса создаётся отдельный SVG-компонент `BrandLogo`, визуально совпадающий с PNG по геометрии, пропорциям, cyan/blue-обводке и мягкому свечению.
- В раскрытом desktop-сайдбаре показывается **только знак**, без надписей «Штаб» и «Тренер у дома».
- На мобильном тот же знак размещается в верхней панели.
- Знак декоративный рядом с уже понятной навигацией, поэтому не должен создавать дублирующее озвучивание screen reader. Ссылка на домашний экран получает доступное имя.

### 2.2. Полное удаление emoji

Emoji удаляются из всех first-party файлов проекта, а не только из видимого интерфейса:

- UI-компоненты;
- trigger-модели и вычисляемые результаты;
- экспорт для Claude;
- CSV-синонимы;
- sanity-страницы и тестовые данные;
- README, актуальное руководство, спецификации и планы;
- текстовые labels и сообщения.

Исключения: сторонние зависимости, Git-история и бинарное содержимое изображений.

Это сознательно меняет обратную совместимость CSV: профильные значения с emoji-префиксом больше не считаются поддерживаемыми синонимами. Поддерживаются текстовые варианты `здоровье`, `форма`, `энергия` и их существующие английские аналоги.

### 2.3. Семантические замены

Emoji не заменяются декоративными картинками. Их смысл распределяется между реальными иконками Lucide, цветом и текстом.

| Область | Представление |
|---|---|
| Тихий триггер | `BellOff`, orange |
| Срочный триггер | `TriangleAlert`, orange + мягкое свечение |
| Средний приоритет | `Clock3`, violet |
| Низкий / информационный | `Circle`, cyan/neutral |
| Сообщение | `MessageCircle` |
| Звонок | `Phone` |
| Тренировка | `Dumbbell` |
| Другое касание | `Ellipsis` |
| Успешный статус | `CircleCheck` + green |
| Выполнить / подтвердить действие | `Check` |
| Ошибка | `CircleAlert` + orange |
| Профиль клиента | цветная точка + текст, без отдельной иконки |

Цвет никогда не является единственным носителем смысла: рядом остаются название статуса, группы или действия.

## 3. Визуальная система

### 3.1. Палитра

Токены брендбука сохраняются:

- `#0E1117` — основной фон;
- `#151923` / `#141925` — стеклянные поверхности;
- `#1B2030` — вложенные строки и поля;
- `#E8ECF4`, `#9AA3B5`, `#5E6678` — уровни текста;
- `#2CE6FF` — единственный primary CTA, активные состояния и focus;
- `#8B5CFF` — вторичный свет и средний приоритет;
- `#36FF9D` — success и профиль «здоровье»;
- `#FF4FD8` — профиль «форма»;
- `#FF9F43` — предупреждения, ошибки и высокий приоритет.

Фон — многослойный градиент, а не однотонная заливка:

1. violet radial glow слева/сверху;
2. cyan radial glow справа/сверху;
3. вертикальный переход от `#121624` к `#0E1117` и `#0A0D13`;
4. `background-attachment: fixed` только там, где это не ухудшает мобильную прокрутку.

### 3.2. Стекло

Основные интерактивные поверхности используют единый рецепт:

- solid fallback `#141925`;
- светлый вертикальный градиент с низкой непрозрачностью;
- border `rgba(255,255,255,.09)`;
- верхний inset highlight;
- многослойная тень;
- `backdrop-filter` как улучшение, не обязательное для читаемости.

В направлении B рабочие карточки получают радиусы 22–30px. Мелкие controls сохраняют радиусы 12–16px, чтобы интерфейс не стал рыхлым.

### 3.3. Типографика

- Display: Gerhaus, uppercase — только H1/H2 и короткие фирменные заголовки.
- Body: Nunito.
- Mono: JetBrains Mono — счётчики, даты, технические подписи, kicker-метки.
- H1 рабочих экранов становится крупнее текущих 27px, но остаётся адаптивным.
- Системный fallback сохраняется до появления локального файла Gerhaus.
- Текущая загрузка Nunito и JetBrains Mono через `next/font` сохраняется.

### 3.4. Движение

- Появление экранных блоков: opacity + translateY, 0.6s, `cubic-bezier(.2,.7,.2,1)`.
- Списки могут использовать короткий stagger, если он не задерживает работу.
- Hover: небольшой подъём и усиление свечения только на устройствах с hover.
- Никаких looping-анимаций, scroll-jacking и тяжёлых layout-анимаций.
- `prefers-reduced-motion: reduce` отключает необязательное движение.

## 4. Навигация и оболочка приложения

### Desktop

- Сайдбар становится самостоятельной стеклянной поверхностью на фоне общего градиента.
- В верхней части — крупный знак NK без wordmark.
- Навигация использует существующие Lucide-иконки.
- Активный пункт получает стеклянную подложку, более яркий текст и спокойное cyan-свечение.
- Кнопка выхода визуально отделена в нижней части.
- TopBar сохраняет действие добавления лида, но визуально становится частью общей оболочки.

### Mobile

- Добавляется компактная верхняя панель со знаком и контекстом текущего экрана.
- Нижний tab bar сохраняется и получает тот же стеклянный рецепт.
- FAB остаётся отдельным быстрым действием, но приводится к форме и свечению primary-кнопки.
- Контент не перекрывается top bar, tab bar или FAB.

## 5. Компонентная архитектура

Изменения строятся поверх существующей структуры, без внедрения новой UI-библиотеки.

### Новые компоненты

- `BrandLogo` — точный SVG-знак, поддерживает только необходимые размеры и декоративное/ссылочное использование.
- `PageHeader` — единая композиция H1, kicker/метаданных и правого действия.
- `IconBadge` — ограниченная обёртка для повторяющейся пары Lucide-иконка + tone.
- `StatusNotice` — success/warning/error/info-блоки с иконкой и текстом.

Компоненты создаются только если реальное повторение подтверждено минимум на двух экранах. Одноразовая разметка остаётся локальной.

### Обновляемые UI-примитивы

- `Button`: единые pill-размеры, glow, hover/active/disabled/loading.
- `Card`: варианты обычного и усиленного стекла.
- `Badge`: цветовая точка, icon slot и текстовая семантика.
- `Input`, `Select`, `Textarea`: единые фон, border, focus, error и disabled.
- `Modal`: усиленная стеклянная поверхность, правильная иерархия действий.
- `EmptyState`: без emoji, с Lucide-иконкой только когда она помогает смыслу.

Не добавляется универсальная конфигурационная система вариантов сверх реально используемых состояний.

## 6. Экраны

Редизайн охватывает:

- `/login`;
- `/register`;
- `/today`;
- `/clients`;
- `/clients/new`;
- `/clients/[id]`;
- `/clients/import`;
- `/dashboard`;
- `/settings`;
- существующие `/dev/*` sanity-страницы.

### «Сегодня»

- Главный showcase-экран направления B.
- Крупный page header, mono-подпись ритуала и счётчик сигналов.
- Один главный cyan CTA — экспорт для Claude.
- Группы триггеров используют Lucide-иконки, tone и текст.
- Строки получают больше вертикального воздуха и чёткое разделение действий.
- Quick touch использует `Check`/`MessageCircle`, а не текстовый Unicode-символ.

### «База»

- Фильтры и список объединяются общей визуальной иерархией.
- Профили — цветная точка + текст.
- CTA добавления клиента остаётся главным только в контексте этого экрана.
- Empty state показывает два действия с одним primary.

### Карточка клиента

- Page header объединяет имя, статусы и основное действие.
- Предупреждение об отсутствии личного факта становится `StatusNotice`.
- Форма и история касаний используют общие стеклянные секции.
- Типы касаний показываются через Lucide.

### Импорт CSV

- Убирается локальный лишний `p-6`, чтобы экран подчинялся общему контейнеру.
- Dropzone получает фирменную стеклянную поверхность и иконку загрузки.
- Preview, ошибки и отчёт используют общие table/status-компоненты.
- Emoji-значения в CSV больше не поддерживаются.

### «Панель»

- Счётчики становятся визуальными metric cards с Lucide-иконками.
- Weekly form и history table получают единую секционную структуру.
- Числа и даты используют mono.
- Сохраняется текущая логика выбора и обновления недели.

### Настройки

- Промпт, пороги и пароль разделяются на отчётливые стеклянные секции.
- Состояния сохранения используют `CircleCheck`, не голый текст или символ.
- Основной CTA остаётся у сохранения настроек; смена пароля — secondary.

### Auth

- Знак становится главным фирменным элементом.
- Карточка входа/регистрации получает более крупную композицию и фон направления B.
- Бизнес-поведение регистрации первого тренера и входа не меняется.

## 7. Данные и бизнес-логика

- Маршруты, Server Components, Server Actions и запросы к Postgres не меняются из-за редизайна.
- Схема БД не требует миграции.
- `emoji` удаляется из runtime-типов trigger result и board view models.
- Trigger kind и priority остаются семантическими enum/string union и определяют UI через mapping на границе представления.
- Экспорт использует текстовые `profileLabel` и `triggerLabel`, без emoji.
- CSV parser больше не содержит emoji-синонимов.

Таким образом, данные остаются семантическими, а выбор иконки полностью принадлежит UI.

## 8. Доступность

- У каждой страницы остаётся один понятный H1.
- Интерактивные icon-only элементы получают `aria-label`.
- Декоративные иконки скрываются через `aria-hidden`.
- Focus-visible сохраняет cyan outline с достаточным контрастом.
- Статусы не кодируются только цветом.
- Toast/status-тексты используют подходящий `aria-live`.
- Проверяется клавиатурная работа модалок и основных форм.
- Route announcement опирается на уникальный H1/заголовок страницы.

## 9. Документация

После реализации обновляются:

- `design-system.md` — фактические токены, компоненты, иконки и запрет emoji;
- `README.md` — favicon/logo и актуальная информация о шрифтах;
- `docs/trainer-guide.md` — названия и внешний вид действий без emoji;
- исторические first-party specs/plans — удаляются emoji-символы без переписывания их смысла.

Временные `.superpowers/brainstorm/*` макеты не являются частью продукта и не коммитятся.

## 10. Проверка

### Автоматическая

1. Тест/скрипт сканирует first-party текстовые файлы и падает при наличии Unicode emoji.
2. Существующие unit/sanity-тесты обновляются под модели без поля `emoji`.
3. TypeScript: `npx tsc --noEmit`.
4. Lint: `npm run lint`.
5. Build: `npm run build`.

### Визуальная

Проверяются desktop и mobile:

- login/register;
- пустой и заполненный `/today`;
- база и фильтры;
- карточка клиента и модалка касания;
- CSV import states;
- dashboard;
- settings.

На каждом экране проверяются:

- ровно один primary cyan CTA;
- отсутствие emoji;
- logo parity с `public/favicon.png`;
- корректные overflow и fixed navigation;
- focus/hover/disabled/loading;
- читаемость иконок и статусов;
- `prefers-reduced-motion`.

### Функциональная регрессия

- вход и регистрация;
- добавление/редактирование клиента;
- quick touch и история касаний;
- экспорт для Claude;
- CSV preview/import;
- weekly stats;
- сохранение настроек и смена пароля.

## 11. Критерии готовности

Редизайн считается завершённым, когда:

1. Все пользовательские экраны соответствуют направлению B.
2. Знак из `favicon.png` используется как favicon и воспроизведён в UI как SVG без текстового локапа.
3. В first-party проекте отсутствуют Unicode emoji.
4. Emoji удалены из данных, экспорта и CSV-совместимости.
5. Бизнес-сценарии работают без изменений.
6. TypeScript, lint, build и профильные тесты проходят.
7. Desktop/mobile визуальная проверка не выявляет перекрытий, потери контраста или несогласованных компонентов.
8. `design-system.md`, README и руководство тренера соответствуют реализации.
